# 技能选择与评估 - 能力验证三法则

## 核心问题

面对众多技能（Skills）和工具（Tools），如何选择？如何避免踩坑？

---

## 一、能力验证三法则

### 法则1：先找现成技能，但不依赖

**核心思想**：不要因为技能包装专业就盲目信任

**验证清单**：
1. [ ] **有没有实现代码？**
   - 不只是 SKILL.md 说明文档
   - 检查 `scripts/` 目录下是否有实际脚本

2. [ ] **能不能本地跑通？**
   - 不要只看文档
   - 必须实际运行测试
   - 确认能完成基本功能

3. [ ] **最近有没有维护？**
   - 检查 GitHub 仓库的 issue 和 PR 活跃度
   - 最后提交时间是否在最近6个月内

**三条都不满足，果断放弃**

### 法则2：API优先原则

**核心思想**：平台官方API往往比第三方技能包更稳定

**优先级排序**：
1. **官方API**（最高优先级）
   - 文档完整，社区活跃
   - 更新及时，bug少
   - 安全可控，权限明确
   - 例如：飞书开放API、微信官方API

2. **稳定第三方库**
   - 有大量用户验证
   - 社区活跃，问题有解答
   - 例如：requests、beautifulsoup4、playwright

3. **技能包**（最低优先级）
   - 新项目，用户少
   - 可能维护不及时
   - 容易踩坑

**实际案例**：
- **飞书文档访问**：官方API → 5分钟搞定 ✅
- **微信文章抓取**：官方API不开放 → 第三方技能包（agent-reach + wechat-article-for-ai）→ MCP超时 → 自己写playwright脚本 ✅

### 法则3：多方案并行

**核心思想**：一个方向卡住时，立刻切换备选方案

**执行策略**：
- **第一优先级**：官方API
- **第二优先级**：playwright自己写
- **第三优先级**：第三方技能包

**时间控制**：
- 每个方案最多投入 30 分钟
- 如果 30 分钟内没跑通，立即切换

## 二、踩坑记录

### 坑1：技能包依赖陷阱

**问题描述**：agent-reach 技能，文档写得非常专业，支持 13+ 平台，但安装后发现只是说明文档，没有实现代码

**原因**：被表面光鲜的文档迷惑，没有检查代码

**解决方案**：
- 安装前检查 `scripts/` 目录
- 运行测试脚本验证功能
- 检查 GitHub 仓库的 commit 历史

**预防措施**：
- 先看代码，再看文档
- 找有实际使用经验的案例

### 坑2：MCP 超时陷阱

**问题描述**：wechat-article-for-ai 技能，用 playwright 抓取微信文章，但 MCP 调用超时，导致失败

**原因**：playwright 首次加载慢，MCP 调用超时时间太短

**解决方案**：
- 自己写脚本，不用 MCP 调用
- 或增加 MCP 超时时间
- 或使用更快的抓取方式

**预防措施**：
- 大型任务不要依赖 MCP
- 自己写脚本更可控

### 坑3：工具选择时间浪费

**问题描述**：在 agent-reach 和 wechat-article-for-ai 上浪费了 30 分钟

**原因**：没有快速验证，盲目信任

**解决方案**：
- 快速验证（5-10 分钟）
- 多方案并行
- 时间控制（每个方案最多 30 分钟）

## 三、工具选择策略

### 1. 官方API优先

**场景**：
- 飞书文档访问
- 微信企业号操作
- 钉钉企业操作

**优势**：
- 文档完整，社区活跃
- 更新及时，bug少
- 安全可控，权限明确

**使用方法**：
```bash
# 飞书 API
curl -X POST "https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal" \
  -H "Content-Type: application/json" \
  -d '{"app_id": "...", "app_secret": "..."}'

# 读取文档
curl "https://open.feishu.cn/open-apis/docx/v1/documents/{token}/blocks" \
  -H "Authorization: Bearer {token}"
```

### 2. Playwright 自己写

**场景**：
- 微信公众号文章抓取
- 需要绕过 Cloudflare 的网站

**优势**：
- 完全可控
- 可以自定义等待时间
- 可以处理复杂页面

**使用方法**：
```python
from playwright.sync_api import sync_playwright

def scrape_wechat_article(url):
    with sync_playwright() as p:
        browser = p.chromium.launch(headless=True)
        page = browser.new_page()
        page.goto(url)
        # 等待页面加载
        page.wait_for_timeout(10000)
        # 获取内容
        content = page.content()
        browser.close()
        return content
```

### 3. 第三方技能包（谨慎使用）

**场景**：
- 没有官方 API 的平台
- 有成熟第三方库的平台

**注意事项**：
- 快速验证（5-10 分钟）
- 检查维护活跃度
- 查看用户反馈

## 四、评估决策框架

### 快速评估（5-10 分钟）

| 维度 | 评分 (1-5) | 说明 |
|------|-----------|------|
| 功能完整性 | | 是否覆盖所有需求？ |
| 文档质量 | | 是否清晰、完整、有示例？ |
| 代码质量 | | 是否有实现代码？代码结构如何？ |
| 维护活跃度 | | GitHub 仓库活跃度如何？ |
| 社区反馈 | | 是否有用户案例？问题解答如何？ |

### 决策树

1. 有官方 API？
   - 是 → 优先使用官方 API
   - 否 → 2

2. 有成熟第三方库？
   - 是 → 快速验证（5-10 分钟）→ 考虑使用
   - 否 → 3

3. 有技能包？
   - 快速验证（检查代码 + 运行测试）→ 考虑使用
   - 不自己写脚本

## 五、时间管理

### 时间分配建议

| 任务类型 | 时间限制 | 说明 |
|---------|---------|------|
| 快速验证 | 5-10 分钟 | 检查代码、运行测试 |
| 深度测试 | 15-30 分钟 | 完整功能测试、性能测试 |
| 文档阅读 | 10-15 分钟 | 阅读文档、理解架构 |
| 代码编写 | 30-60 分钟 | 自己写脚本 |

### 超时控制

- 每个方案最多投入 30 分钟
- 超时立即切换备选方案
- 记录踩坑经验，避免下次重复

## 六、经验总结

### 核心教训

1. **不要被表面光鲜的文档迷惑**
   - 先看代码，再看文档
   - 检查 GitHub 仓库活跃度
   - 查找实际使用案例

2. **官方API永远比第三方稳定**
   - 文档完整
   - 社区活跃
   - 更新及时
   - bug少

3. **多方案并行，时间控制**
   - 第一优先级：官方API
   - 第二优先级：自己写脚本
   - 第三优先级：第三方技能包
   - 每个方案最多 30 分钟

4. **快速验证，果断放弃**
   - 不满足三条验证法则，果断放弃
   - 不要在无用的技能上浪费时间
   - 记录踩坑经验，避免下次重复

### 推荐工作流

```
1. 需求分析
   ↓
2. 搜索现有方案（官方API/第三方库/技能包）
   ↓
3. 快速验证（5-10 分钟/方案）
   ↓
4. 选择最优方案
   ↓
5. 深度测试（15-30 分钟）
   ↓
6. 记录经验（成功/失败）
```
